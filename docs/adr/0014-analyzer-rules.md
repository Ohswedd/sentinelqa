# ADR-0014: Analyzer rules (categorization, root cause, repro, retry, optional LLM)

## Status

Accepted

<!-- Authors: @ohswedd -->

## Context

ships the Analyzer module : the layer between the Runner and the Reporter ( / 15), Healer, and Scorer. For every failed test the Analyzer must answer four questions, in order:

1. **What kind of failure is this?** the documentation enumerates ten canonical categories (`app_bug`, `test_bug`, `environment_failure`, `flake`, `data_setup_failure`, `auth_failure`, `api_failure`, `performance_regression`, `security_finding`, `accessibility_violation`) plus an honest `unknown` bucket for novel patterns.
2. **Why do we think so?** The Reporter and SDK both need a short, plain-English root-cause hypothesis with a confidence score and pointers to the evidence (trace, HAR, network log) that supports it.
3. **How do I reproduce it?** Developers need copy-pasteable steps that do not leak credentials, and an optional minimal Playwright spec they can adapt for a regression test.
4. **Should the runner retry, or is this stable?** our published policy-loops: retry transient failures, never auto-retry deterministic ones, and surface stable test-side defects as quarantine candidates for human review.

Several design pressures shaped the answer:

- **Determinism first.** (Planner) and (Generator) shipped LLM adapters behind feature flags but kept their deterministic cores authoritative. The Analyzer must do the same — every result must be reproducible without an API key and without a network call.
- **The runner already aggregates per-test signals.** `engine.runner.results.RunnerOutcome` (ADR-0013) carries each `TestExecution`'s status, retries, evidence paths, and error message. The Analyzer should consume that contract — not re-parse the JSONL stream.
- **Module-level errors need categorization too.**'s `RunLifecycle.run_modules` wraps every module call in a broad `except Exception`. The implementation just stuffed `type(exc).__name__: str(exc)` into `ModuleOutcome.error_message`. The Reporter and SDK have nowhere to read _why_ the module fell over — was it a missing dep, a network glitch, or a malformed suite? has to close this gap.
- **The LLM is a refinement, not a replacement.** When an OpenAI / Anthropic key is configured, the Analyzer may append a single sentence of context to the deterministic hypothesis. It must never replace the category, hypothesis, or confidence. Budget enforcement and locked prompts mirror the Planner adapter (ADR-0011) so the cost model and review process are consistent.

## Decision

We adopt the following conventions for the Analyzer module:

1. **Categorization is a small ordered rule set with no hidden state.** `engine.analyzer.categorize.categorize(signal)` runs eleven named rules (`_rule_flake`, `_rule_browser_crash`, `_rule_fixture_auth_failure`, `_rule_app_5xx`, `_rule_api_4xx`, `_rule_auth_4xx`, `_rule_locator_timeout`, `_rule_a11y`, `_rule_security`, `_rule_performance`, `_rule_data_setup`). Every rule that matches is recorded — the highest-confidence match becomes the primary `FailureClassification.category`; the rest become `secondary`. Rules deliberately under-claim ("Locator timed out _while the app responded successfully_") so the Healer never auto-fixes against a noisy signal. Signals that match no rule return `("unknown", 0.3, "no rule matched; inspect the trace")` — never a fake match.
2. **Hypothesis = category template + evidence snippet, capped at 1024 chars.** `engine.analyzer.root_cause.hypothesize(signal, classification)` looks up a per-category template (e.g. `"The server returned an error ({0}) while the test was exercising the app. Likely cause:..."`), interpolates a redaction-safe snippet (`"HTTP 502 from https://example.test/api/orders"`), and emits a `RootCauseHypothesis` with the deterministic confidence the rule set produced (never re-inflated) and a fixed `next_actions` checklist per category (e.g. `"Open the failing step in the trace viewer"`). URLs are stripped of query/fragment and clipped at 80 chars before they appear in the snippet so secrets in query strings never leak. The hypothesis text is hard-clipped at 1024 chars regardless of input size.
3. **Reproduction steps are evidence-grounded and credential-free.** `engine.analyzer.repro.reproduction(signal, auth_env_vars=..., base_url=...)` returns an ordered tuple of human-readable steps: open the trace, visit the route, authenticate via the configured `*_env` names (e.g. `$SENTINEL_TEST_USER` — never literal credentials, per the engineering guidelines), replay each captured `StepRecord`, and observe the expected-vs-actual outcome. `build_repro_spec(signal, base_url, finding_id, auth_env_vars=...)` emits a minimal Playwright TS spec that starts with the locked `// SENTINELQA AUTO-GENERATED REPRO SPEC` banner; auth, if present, is pulled from `process.env["NAME"]` and never inlined. The spec is intentionally a starting point, not a self-healing replay — the user replaces the TODO assertion with the concrete failed assertion.
4. **Retry / quarantine is deterministic, capped at two auto-retries, and history-aware.** `engine.analyzer.retry_decision.should_retry(signal, classification, history=None)` returns a typed `RetryDecision` (`retry` | `quarantine_candidate` | `no_action`). The recommendations follow the rules: - `flake`, `environment_failure` → `retry` (transient by definition). - `auth_failure` in the fixture → `no_action` (config-bound; retrying re-runs the same fixture). - `auth_failure` mid-test → `retry` (distinguish session timeout from a true block). - `app_bug`, `api_failure`, `performance_regression`, `security_finding`, `accessibility_violation`, `data_setup_failure` → `no_action` (deterministic). - `test_bug` → `no_action` on first attempt; `quarantine_candidate` once we've already retried _or_ once optional `FailureHistory` shows recurring failures against a healthy app (≥ 3 of the last few runs failed). - Regardless of category, two auto-retries is the hard cap. Above that the rule returns `no_action` with a "conservative auto-loop boundary" reason. the engineering guidelines, not the runner.
5. **Module-error path is rehomed through the analyzer.** `engine.orchestrator.run_lifecycle.run_modules` now calls `engine.analyzer.categorize.categorize_module_error(module=..., exc_type=..., exc_message=...)` inside each `except` arm and attaches the result to `ModuleOutcome.error_category` / `error_confidence` / `error_rationale` / `error_type`. The catch-all defaults to `environment_failure` at low confidence (we'd rather under-claim than mis-blame the app). Import / network / safety / config exceptions get higher confidence; `TestExecutionError` is classified as `test_bug`. The Reporter and SDK can now surface _why_ a module failed instead of just "errored".
6. **LLM explainer mirrors the Planner adapter exactly and is off by default.** `engine.analyzer.llm_explainer` defines an `LlmExplainer` Protocol, a `NullLlmExplainer` default, an `HttpLlmExplainerBase` for shared budget / cost / API-key wiring, and two HTTP-only provider adapters (`OpenAiLlmExplainer`, `AnthropicLlmExplainer`). Vendor SDKs are not imported — we POST directly via `httpx` (same rationale as ADR-0011). The locked prompt lives at `engine/analyzer/llm_prompts/explainer.v1.md`; bumping it requires a new ADR per the engineering guidelines`{ "refinement": "<= 400 chars" }`; malformed responses return `None` (analyzer keeps its deterministic hypothesis). Budget is bounded by `analyzer.llm.max_usd_per_run` (default $0.25), enforced by the shared `engine.planner.llm_adapter.ensure_within_budget`. With the feature flag off (default), no LLM is constructed, no HTTP client is opened, and analyzer output is fully deterministic.

## Consequences

- **Positive:** - Every failure now produces a typed `AnalyzerResult` with category, hypothesis, repro steps, and retry decision — all four the documentation outputs in one place. The Reporter and Healer consume the same record; no duplicate inference paths. - Categorization rules are small, ordered, and individually unit-testable. Adding a new pattern (e.g. a CSP-specific signal) means adding one `_rule_*` function plus its test — no flow refactor. - Repro output is credential-free by construction. The spec exporter accepts `auth_env_vars` and writes `process.env["NAME"]`; the plain-text repro likewise uses `$NAME`. the engineering guidelines, not via post-hoc redaction. - Retry decisions are bounded: at most two auto-retries per test, always with a rationale the user can read. CI loops cannot accidentally infinite-retry an `auth_failure` because the auth fixture is configured wrong. - Module-level errors are now categorized, so a " fall-through" no longer surfaces as a mystery `errored: RuntimeError`. The Reporter (and SDK) can show _why_ a module fell over with the same vocabulary the per-test analyzer uses. - The LLM explainer is opt-in and never authoritative. Running `make ci` in an air-gapped environment still produces complete analyzer output; the LLM only adds context when enabled and within budget.
- **Negative / trade-off:** - The categorization rules rely on text-pattern matching against `error_message` / `error_name`. Playwright's error strings are fairly stable but not contract-versioned; a Playwright major bump could require a rule sweep. The phase mitigates this with broad fallbacks (`unknown` rather than mis-categorize). - The repro spec is a TODO scaffold, not a turnkey replay. Step-text → concrete-selector translation is intentionally not attempted because the analyzer cannot verify the selector against the live DOM. The user finishes the spec. - Module-error classifications expand `ModuleOutcome` (in-memory) with four optional fields. The wire format used by run.json is not changed in this phase; will surface these in the HTML report. Until then they live in-memory and in the lifecycle's `module_outcomes` list. - The optional LLM explainer adds two new dependencies to the dependency surface — both already present from (planner). No new transitive deps.
- **Follow-up obligations (already planned, not deferred from this phase):** - (Quality scoring) reads `AnalyzerResult.classification.confidence` and the retry decisions to apply policy gates (`block_on_critical`, `max_flake_rate`, quarantine fold-down). - (HTML & JSON Reports) renders the per-failure hypothesis, repro steps, and module-level categorization in the HTML report. - (Healer) gates its auto-repair on `analyzer.confidence >= threshold` and uses `next_actions` to choose a repair template; it never bypasses the deterministic category. - Bumping `PROMPT_VERSION` in `engine/analyzer/llm_explainer.py` requires a new ADR per the engineering guidelines, identical to the Planner prompt policy (ADR-0011).

## Alternatives considered

- **Single "failure" category with an open-ended `cause: str` field.** Rejected because it pushes parsing onto the Reporter / Healer / SDK — exactly the work the analyzer exists to do.
- **Letting the runner emit `RetryDecision` directly so the analyzer disappears.** Rejected because retry is one of four outputs (category, hypothesis, repro, retry); collapsing into the runner would re-couple Playwright execution with classification rules. ADR-0013 deliberately kept those concerns apart.
- **Always running the LLM and using its output as the primary hypothesis.** Rejected for the same reason ADR-0011 rejected LLM-only planning: it would make CI non-deterministic, vendor-coupled, and unsafe to run in air-gapped environments. The LLM stays a refinement.
- **Auto-fixing test bugs (e.g. locator updates) from the analyzer.** Rejected — that is (Healer)'s job, gated on the analyzer's confidence and the user's review policy.
