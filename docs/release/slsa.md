# SLSA build provenance

Starting with v1.7.0 (phase 37), every published artifact carries a
signed [SLSA build provenance](https://slsa.dev/spec/v1.0/provenance)
statement generated by GitHub's
[`actions/attest-build-provenance@v2`](https://github.com/actions/attest-build-provenance)
action and signed via Sigstore.

| Surface        | Artifact                  | Attestation mechanism                                                           | SLSA level |
| -------------- | ------------------------- | ------------------------------------------------------------------------------- | ---------- |
| PyPI           | wheels + sdists           | `attest-build-provenance` + PyPI Trusted Publisher OIDC                         | L3         |
| npm            | tarball                   | `attest-build-provenance` + npm provenance                                      | L3         |
| Docker Hub     | runner image              | `attest-build-provenance` (OCI referrer) + Buildx `provenance: mode=max` + SBOM | L3         |
| GitHub Release | wheels + sdists + tarball | `attest-build-provenance` over every release asset                              | L3         |

The build-track gets us to L3 today. L4 (hermetic, two-party reviewed
builds) is the next milestone — see "Path to L4" below.

## Verifying a release

For any release tag `vX.Y.Z`:

```bash
# Python wheel from PyPI
gh attestation verify \
  --owner Ohswedd \
  sentinelqa_cli-X.Y.Z-py3-none-any.whl

# npm tarball
gh attestation verify \
  --owner Ohswedd \
  sentinelqa-ts-runtime-X.Y.Z.tgz

# Docker image (digest from `docker manifest inspect`)
gh attestation verify \
  --owner Ohswedd \
  oci://docker.io/sentinelqa/runner@sha256:<digest>
```

The npm + PyPI registries also expose their own provenance UIs; both
should agree with the GitHub-side attestation. A mismatch is a
release-engineering bug — open an issue and do not consume the
artifact.

## Path to L4

L4 requires:

1. **Hermetic builds.** Every input to the build step must be pinned
   and reproducible: pinned base images (digest, not tag), pinned
   `uv` / `pnpm` versions, no network access at build time beyond the
   pinned dependency lock. Today PyPI / Docker builds use frozen
   lockfiles but pull from network during the build step; the
   reproducibility audit is incomplete.
2. **Two-party review.** The publish workflows already gate on the
   `pypi-release` / `npm-release` / `docker-release` GitHub
   Environments, which require a single human approver. L4 wants
   two-party approval — see the runbook for the planned change.
3. **Hermetic build infrastructure.** GitHub-hosted runners are
   ephemeral but not hermetic by SLSA's definition. Moving release
   builds to a self-hosted runner pool with no inbound network is the
   path; tracked in ADR-0048 (planned).

When all three land, this page bumps the L3 entries above to L4.
