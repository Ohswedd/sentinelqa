# SentinelQA — Definition-of-Done Sweep (Phase 29.08)

## Scope

CLAUDE.md §18 lists the closing conditions every contribution must satisfy:

1. Implementation matches PRD.
2. Tests exist and pass.
3. Types / lint pass.
4. Safety reviewed.
5. Reports / schemas updated.
6. Docs / PRD updated.
7. No secrets introduced.
8. `git status` clean after commit.

Phase 29.08's deliverable is a single command — `make dod` — that runs the
local equivalent of the phase-gate review so a contributor gets the same
verdict the reviewer will.

## `make dod` shape

```
dod: ci
    pytest tests/integration/release/test_secret_leak.py    # 29.02 gate
    pytest tests/integration/release/test_determinism.py    # 29.03 gate
    git status --porcelain                                  # CLAUDE §18.8
    -> green if all pass and tree is clean
```

`ci` already covers `format-check`, `lint`, `typecheck`, `adr-check`, and
the full pytest + vitest test sweep. The release-tier `test_secret_leak.py`
and `test_determinism.py` extend the gate with the recurring audit checks
introduced in Phases 29.02 and 29.03.

## Coverage gate

Per CLAUDE.md §17, `make coverage` must remain green at the 95 % floor
specified in `pyproject.toml`. Phase 29 discovered a latent
`coverage combine` failure where child Python processes started inside
`tmp_path` (CLI smoke, generator `tsc` acceptance, redaction parity)
initialised their own coverage in statement-mode while the parent ran in
branch-mode, causing the combine step to abort with
`Can't combine statement coverage data with branch data`.

Fix shipped in this phase: `tests/conftest.py` now seeds
`COVERAGE_RCFILE` + `COVERAGE_FILE` env vars to absolute repo paths, so
every child Python process picks up the same `[tool.coverage.run]`
configuration regardless of cwd. With that in place:

```
$ uv run pytest --cov --cov-report=term
…
TOTAL                                                                 13365    430   3536    353    95%
Required test coverage of 95.0% reached. Total coverage: 95.13%
=============== 3027 passed, 2 skipped, 27 deselected in 48.16s ================
```

Coverage gate green at 95.13 % (floor 95 %).

## How `make dod` is used

- Local: a contributor runs `make dod` before pushing. Output is a green
  banner or a precise failure pointing at the broken gate.
- Phase-gate review: the reviewer pastes the `make dod` output into the
  phase row in `STATUS.md` to evidence the gate.
- CI: `make ci` is the GitHub Actions surface; `make dod` is the local
  superset. We don't run `make dod` in CI because CI already runs the
  three components individually (ci + the two release tests) and the
  `git status` clean check is a property of the committed PR.
